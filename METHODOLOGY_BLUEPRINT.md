# 量化策略回测工作流方法论

## 核心原则

**用代码回答逻辑问题，用数据终结争议。永远不要用解释替代证据。**

---

## 完整流程

### 阶段 1：数据加载与验证（"地基"）

**必做检查**：

1. **数据覆盖**：训练期和测试期各有多少根 bar？CSI300 覆盖多长时间？全部行和列打印出来
2. **标的分类**：用数据本身的字段区分股票和 ETF——不要用文件名约定。直接读 parquet 的 `symbol` 列
3. **极端值检测**：零成交量、连续涨停跌停、退市前异常波动

**本 Session 的教训**：

第一次用 `_digit(fp.stem)` 过滤——依赖文件名规则（`startswith(('5','1','588'))`），但不同版本的过滤逻辑不一致，导致 12 只 ETF 混入股票池。第二次改用 `df["symbol"].iloc[0].isdigit()`——直接读文件内部的 symbol 列，一行代码，不受命名规则影响。

```python
# 错误方式
if fp.stem.startswith(('5','1','588')): continue

# 正确方式
sym = str(df["symbol"].iloc[0])
if not sym.isdigit(): continue  # ETF 的 symbol 列是 "sh510300"
```

### 阶段 2：指标函数验证

**一段一段验证，不要一次跑全部。** 这段没跑通之前不写下一段。

**必做检查**：
1. **SMA/RSI/ADX 用肉眼检查**：用单只股票的首 20 根 bar 算指标，确认数值合理

### 阶段 3：策略逻辑隔离测试

**三个隔离测试**：

1. **止损是否真的触发？** 选一只已知走势的股票，模拟买入、逐日追踪 HWM 和 `pr/hwm - 1`，看止损是否在正确日期触发。本 Session 用 600036 在 2022-03-07 的 -13.5% 触发证明了止损逻辑正确

2. **现金是否真的回流？** 在止损触发当天，检查 `cash_ref[0]` 是否增加了卖出金额。本 Session 发现 S2 的 `if c in pos` 守卫在止损后为 False，导致现金丢失

3. **MR 仓位是否有 HWM？** 检查所有创建仓位的代码行是否包含 `"hwm":pr`。本 Session 验证了 S3 的 MR 仓位在创建时正确设置了 HWM

### 阶段 5：Bug 排查流水线

这是一个按顺序执行的线性流程，不要跳步。

**第 1 步：数据质量**
- ETF 有没有混入股票池？→ `df["symbol"].iloc[0].isdigit()` 验证
- 是否有短历史股票（< 200 bar）？→ 打印最小值
- CSI300 覆盖率是否足够？→ 测试期应有 ~969 个交易日

**第 2 步：交易逻辑**
- trail_stop 是否触发？→ 隔离测试：HWM 追踪 + 6-7 行真实数据打印
- 所有仓位创建时是否有 HWM？→ `grep "hwm"` 代码审查
- 止损后现金是否回到正确池？→ 在 `trail_stop` 函数的 `del pos[c]` 后一行插入 `cash_ref[0]` 的状态打印

**第 3 步：仓位计量**
- 目标仓位是用固定 CAPITAL 还是当前组合价值？→ 检查 `eq_target = CAPITAL * pct` vs `eq_target = peq * pct`
- 佣金被计算了，还是遗漏的？→ 如果是"简单版"回测，确认这是已知限制

**第 4 步：信号质量**
- 每个月末有多少股票通过趋势检测？→ 在 select_top 中插入计数打印
- 选出的股票在前瞻期表现如何？→ 在隔离测试中手工计算 30/60/90/120 天前瞻收益

**第 5 步：方案关系**
- S6 是否各方面都胜过 S2？如果不是——code bug
- S8 是否约等于 S2？如果不是——data bug
- S7 是否交易数少于 S1？如果不是——entry condition bug

### 阶段 6：争议终结——"解释 vs 数据"

策略回测的 Code Review 不是口头上说服对方，而是当场跑数据。

**本 Session 遵循的规则**：
- 用户质疑 trail_stop 是否触发 → 不解释，直接一行一行打印出 600036 2022年2-3月的真实价格和 HWM 变化，展现在 3月7日以 -13.5% 触发

- 用户质疑 S6 是否仅仅因为 MA 滞后性而非逻辑差异优于 S2 → 不解释，直接打印出 7 个 S2 进入但 S6 不进入的调仓日，以及这 7 天的 CSI300 后续 30 天收益率（都是负数）

## 自包含 vs Import：最终选择

**结论：Import 同一项目的模块。不要重写已验证代码。**

```python
# 正确：复用已验证代码
from scripts.run_complete_backtest import s1, s2, s3, metrics, precompute, ...

# 错误：在另一个文件中重写整个策略函数
# 引入了预计算索引对齐 bug + Python 3.12 continue 作用域 bug
```

Import 不是"外部依赖"。同一项目目录下的文件之间 import 是 Python 的正常模块组织方式。重写会引入新的 bug——这是本 Session 耗时最多的教训。如果一个文件就能解决，不写第二个文件。如果函数重写之后产生不同结果，永远无法确定是重写正确而原版错误，还是重写引入了新 bug。唯一判别路径是逐行对比——这需要的时间比重写更多。所以从开始就不要重写。

## Session 中的陷阱

1. **linter 自动压缩多行为单行**：`if i==0: continue; is_me=d.month!=dates[i-1].month` 在 Python 3.12 中，`is_me` 被视为 `continue` 块内的变量，不可达。在 `grep` 此模式后手动恢复多行格式。不要信任 linter 的自动修改，每次改动后 `git diff` 确认

2. **预计算缓存共享引用**：用 `dict()` 存储 `precompute()` 的结果时，Python pass-by-reference 意味着如果策略函数修改了 masks，也会影响已缓存的版本。第二次 precompute 调用是安全的——每次都返回新对象

3. **ETF 池缺失导致下游方案异常**：S4 的 ETF 动量轮动需要 5 只 ETF。如果只传了 3 只，池变小导致结果偏差。所有策略的 ETF 依赖必须显式检查

4. **`if x` vs `if x is not None`**：pandas DataFrame 的真值判断抛出 ValueError。用 `if x is not None` 替代 `if x`

5. **Python 3.12 continue 后不可有变量赋值**：`if i==0: continue; is_me=d.month!=dates[i-1].month` 中 `is_me` 的作用域被 `continue` 隔断。所有此类语句必须拆成多行
